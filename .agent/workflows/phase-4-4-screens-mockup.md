# Phase 4.4: 画面モック生成ワークフロー

## 目的・概要

このフェーズでは、**ユーザー主導**でClaude Code（CC）にHTMLベースのデザインモックを生成させる。

AGの作業はPh4.3で完了しており、このフェーズはユーザーとCCの連携で進行する。

---

## 担当者

| 役割 | 担当 |
|------|------|
| 主導 | **User（施主）** |
| HTML生成 | CC（職人） |
| アセット生成 | AG（画像生成依頼時のみ） |

---

## 前提条件

- [ ] Phase 4.3 が完了していること
- [ ] `docs/design/screens/index.md` が存在すること
- [ ] `PRODUCT.md` の Current Phase が Phase 4.4 であること

---

## ワークフロー

### Step 1: CCへのHTMLモック生成依頼（ユーザー実行）

ユーザーがCCに以下のプロンプトで依頼する：

```
docs/design/screens/index.md と docs/tech-stack.md を読み込んで、
HTMLベースのデザインモックを生成してください。

## 事前に読み込むドキュメント

以下の順番で読み込んでください：
1. `docs/requirements.md` - 要件定義
2. `docs/design/basic-design.md` - 基本設計
3. `docs/design/screens/index.md` - 画面設計仕様
4. `docs/tech-stack.md` - 実装技術

## 重要な認識

このHTMLプロジェクトは**デザインモック**であり、実際の実装は
`docs/tech-stack.md` に記載された技術で行います。
そのため、対象技術で実装しやすい構成を心がけ、
コードにはヒントとなるコメントを入れてください。

## 生成ルール

1. **出力先**: docs/design/screens/html/
2. **構成**:
   - 全画面を1つのHTMLプロジェクトとして作成
   - index.html をエントリーポイントに
   - 画面間の導線を設定（静的リンク）
3. **CSS設計**:
   - TailwindCSS風のユーティリティクラス
   - 全画面でトンマナを統一
4. **レスポンシブ**:
   - PC/スマホ対応
   - index.mdのブレイクポイントに従う
5. **デザイン**:
   - index.mdのデザイン方針に従う
   - frontend-designスキルを活用し、独自性のある印象的なデザインを
   - 具体的なカラーコードやフォントはあなたが決定してよい
6. **動的機能**:
   - 不要（静的なデザイン確認用）
   - JavaScriptは最小限（ハンバーガーメニュー等のみ）
7. **言語**:
   - 日本語で表記
8. **スキル**:
   - 必ず frontend-design スキルを使用すること

## アセットについて

画像などのアセットは後から私が追加します。
今はアセットが無くても404エラーを気にせず、プレースホルダーを配置してコーディングしてください。

## 出力物

1. **HTMLプロジェクト**: docs/design/screens/html/ 配下
2. **README.md**: HTMLプロジェクトのルートに作成し、以下を記載
   - 採用したデザイン設計の説明（カラーパレット、フォント、デザインコンセプト等）
   - 各画面の概要
   - 特記事項

## アセット依頼

アセットが必要な場合は、以下のフォーマットで必要なアセットをすべてリストアップしてください：

## 必要なアセット一覧

### 1. [用途を説明]
- ファイル名: xxx.png
- サイズ: 000x000px
- 形式: PNG/JPG/SVG
- 説明: 具体的な説明（色、雰囲気、含むテキスト等）
```

---

### Step 2: CCによるHTMLモック生成

CCは以下を実行する：

1. `docs/design/screens/index.md` を読み込む
2. `docs/tech-stack.md` を読み込む
3. `frontend-design` スキルを使用
4. `docs/design/screens/html/` ディレクトリを作成
5. 全画面のHTMLファイルを生成
6. 共通CSSファイルを生成
7. HTMLプロジェクトのルートに `README.md` を作成
8. 必要なアセットがあれば、フォーマットに従って依頼をレスポンス

### CCの出力例

```
docs/design/screens/html/
├── README.md            # デザイン設計の説明
├── index.html           # ホーム画面
├── login.html           # ログイン画面
├── dashboard.html       # ダッシュボード
├── ...
├── css/
│   └── style.css        # 共通スタイル
├── js/
│   └── main.js          # 最小限のJS（メニュー開閉等）
└── assets/
    └── (プレースホルダー)
```

---

### Step 3: アセット生成依頼（必要な場合）

CCからのアセット依頼を受け取ったら、ユーザーはAGに依頼する。

#### アセット依頼の例

```
CCから以下のアセット依頼がありました。生成してください。

## 必要なアセット一覧

### 1. メインロゴ
- ファイル名: logo.png
- サイズ: 200x50px
- 形式: PNG（透過）
- 説明: ヘッダーに配置するメインロゴ。モダンでミニマルな雰囲気。青系統。テキスト「AppName」を含む。

### 2. ヒーロー背景画像
- ファイル名: hero-bg.jpg
- サイズ: 1920x1080px
- 形式: JPG
- 説明: トップページのヒーローセクション背景。抽象的なグラデーション、または関連するイメージ。
```

#### AGの対応

1. 画像生成ツール（generate_image）を使用してアセット生成
2. 指定の出力先（`docs/design/screens/_assets/`）に保存
3. 完了をユーザーに報告

---

### Step 4: デザイン確認・修正

ユーザーがHTMLモックを確認し、修正が必要な場合はCCに依頼する。

#### 修正依頼の例

```
docs/design/screens/html/dashboard.html のヘッダーを修正してください。

## 修正内容
- ロゴのサイズを大きくする（現在の1.5倍）
- ナビゲーションの間隔を広げる
```

---

### Step 5: 完了確認

全ての画面とアセットが完成し、ユーザーが承認したらフェーズ完了。
